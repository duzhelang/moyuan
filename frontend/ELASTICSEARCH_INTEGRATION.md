# Elasticsearch 集成评估

## 当前搜索实现分析

### 现有实现
- 使用 MySQL LIKE 查询进行全文搜索
- 搜索接口：`/search?keyword=xxx`
- 搜索范围：诗词、诗人、帖子

### 优点
1. 实现简单，无需额外基础设施
2. 维护成本低
3. 适合小数据量场景

### 缺点
1. **性能问题**：数据量大时（>10万条），LIKE 查询效率低下
2. **功能限制**：
   - 不支持分词
   - 不支持相关性排序
   - 不支持高亮显示
   - 不支持模糊匹配
3. **扩展性差**：无法支持复杂的搜索需求

## Elasticsearch 集成方案

### 方案一：轻量级集成（推荐）

**适用场景**：数据量在 10-100 万条，搜索需求相对简单

**实现步骤**：
1. 后端集成 Elasticsearch 客户端
2. 创建索引映射
3. 实现数据同步（监听数据库变更）
4. 修改搜索 API 使用 ES 查询

**索引结构设计**：

```json
{
  "mappings": {
    "properties": {
      "id": { "type": "integer" },
      "type": { "type": "keyword" },  // poem/poet/post
      "title": { 
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart"
      },
      "content": { 
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart"
      },
      "author": { "type": "keyword" },
      "dynasty": { "type": "keyword" },
      "category": { "type": "keyword" },
      "createTime": { "type": "date" },
      "viewCount": { "type": "integer" },
      "likeCount": { "type": "integer" }
    }
  }
}
```

### 方案二：完整集成

**适用场景**：数据量 > 100 万条，需要高级搜索功能

**额外功能**：
- 搜索建议（自动补全）
- 同义词扩展
- 搜索统计分析
- 个性化搜索结果

## 实施建议

### 阶段一：准备阶段
1. 评估当前数据量和增长趋势
2. 确定搜索功能需求优先级
3. 准备 Elasticsearch 环境（开发/测试/生产）

### 阶段二：后端集成
1. 添加 Elasticsearch 依赖
2. 配置连接参数
3. 创建索引管理服务
4. 实现数据同步机制
5. 修改搜索 API

### 阶段三：前端适配
1. 更新搜索结果展示（支持高亮）
2. 添加搜索建议功能
3. 优化搜索体验

### 阶段四：测试与优化
1. 功能测试
2. 性能测试
3. 压力测试
4. 监控告警配置

## 成本评估

### 基础设施成本
- Elasticsearch 集群：至少 3 个节点
- 存储空间：根据数据量预估
- 内存：每个节点至少 8GB

### 开发成本
- 后端开发：2-3 周
- 前端开发：1 周
- 测试：1 周

### 维护成本
- 日常运维
- 索引优化
- 版本升级

## 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 数据同步延迟 | 搜索结果不一致 | 实现双写机制，异步同步 |
| ES 集群故障 | 搜索功能不可用 | 降级到 MySQL 查询 |
| 性能不达预期 | 用户体验差 | 优化索引结构，增加缓存 |

## 结论

**当前阶段建议**：
- 如果数据量 < 10 万条，继续使用 MySQL LIKE
- 如果数据量在 10-100 万条，实施轻量级集成
- 如果数据量 > 100 万条，实施完整集成

**优先级**：中低（可与其他功能并行开发）

**建议时间窗口**：下个版本迭代时考虑
