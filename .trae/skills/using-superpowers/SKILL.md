<extremely-important>
如果你认为哪怕有1%的概率某个技能可能适用于你正在做的事情，你绝对必须调用该技能。

如果一个技能适用于你的任务，你没有选择。你必须使用它。

这不是可谈判的。这不是可选的。你不能为此找借口。
</extremely-important>

## 指令优先级

超级能力技能覆盖默认系统提示行为，但用户指令始终优先：

1. 用户的明确指令（最高优先级）
2. 超级能力技能（覆盖默认系统行为）
3. 默认系统提示（最低优先级）

## 如何使用技能

使用 `Skill` 工具。当你调用一个技能时，其内容被加载并呈现给你——直接遵循它。永远不要用 Read 工具读取技能文件。

## 规则

**在任何响应或操作之前调用相关或请求的技能。** 哪怕有1%的概率技能可能适用，你都应该调用技能来检查。如果调用的技能对情况来说是错误的，你不需要使用它。

## 技能优先级

当多个技能可能适用时，使用以下顺序：

1. **过程技能优先**（brainstorming, debugging）——这些决定如何接近任务
2. **实现技能其次**——这些指导执行

"让我们构建X" → 先 brainstorming，然后实现技能。
"修复这个bug" → 先 debugging，然后领域特定技能。

## 技能类型

**刚性**（TDD、调试）：精确遵循。不要调整掉纪律。
**灵活**（模式）：将原则适配到上下文。

技能本身会告诉你是哪种类型。

## 红旗

这些想法意味着停止——你正在找借口：

| 想法 | 现实 |
|---------|---------|
| "这只是一个简单问题" | 问题是任务。检查技能。 |
| "我需要更多上下文" | 技能检查在澄清问题之前。 |
| "让我先探索代码库" | 技能告诉你如何探索。先检查。 |
| "这不值得用正式技能" | 如果技能存在，使用它。 |
| "我记得这个技能" | 技能会演进。阅读当前版本。 |
| "技能是大材小用" | 简单的事会变复杂。使用它。 |
| "我先做这一件事" | 做任何事之前先检查。 |
