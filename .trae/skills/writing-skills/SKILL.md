# 编写技能

## 概述

编写技能是将TDD应用于过程文档。

**核心原则：** 如果你没有观察到没有该技能时代理的失败，你就不知道技能是否教了正确的东西。

## 什么是技能

技能是经过验证的技术、模式或工具的参考指南。技能帮助未来的AI实例找到并应用有效的方法。

技能是：可重用的技术、模式、工具、参考指南
技能不是：关于如何解决一次问题的叙述

## 何时创建技能

创建时机：
- 技术对你来说不是直观明显的
- 你会在跨项目中再次引用它
- 模式广泛适用（非项目特定）
- 其他人会受益

不创建时机：
- 一次性解决方案
- 其他地方已充分记录的标准做法
- 项目特定约定（放入项目配置文件）
- 机械约束（如果可以通过验证自动化，自动化它——把文档留给需要判断的地方）

## SKILL.md 结构

元数据：
- 两个必需字段：`name` 和 `description`
- `name`：只使用字母、数字和连字符
- `description`：第三人称，仅描述何时使用（而非做什么）
  - 以"Use when..."开头以聚焦触发条件
  - 包含具体症状、情况和上下文
  - **绝不总结技能的过程或工作流**
  - 保持500字符以内

```markdown
---
name: skill-name-with-hyphens
description: 当 [具体触发条件和症状] 时使用
---

# 技能名称

## 概述
这是什么？核心原则用1-2句话。

## 何时使用
症状和用例列表
何时不使用

## 核心模式
之前/之后的代码对比

## 快速参考
扫描常见操作的表格或列表

## 实现
简单模式的内联代码
链接到文件用于重型参考或可重用工具

## 常见错误
什么问题 + 修复

## 实际影响（可选）
具体结果
```

## Claude搜索优化 (CSO)

### 1. 丰富的描述字段

描述应该只描述触发条件。不要总结技能的过程或工作流。

为什么：当描述总结技能的工作流时，AI可能会遵循描述而不是阅读完整技能内容。

```yaml
# ❌ 坏：总结工作流
description: 执行计划时使用 - 为每个任务调度子代理并在任务间进行代码审查

# ❌ 坏：过多过程细节
description: 用于TDD - 先写测试，观察失败，写最小化代码，重构

# ✅ 好：仅触发条件
description: 当执行具有独立任务的实现计划时使用
```

### 2. 关键词覆盖

使用AI会搜索的词汇：
- 错误信息
- 症状："flaky"、"hanging"、"race condition"
- 同义词
- 工具：实际命令、库名称、文件类型

### 3. 描述性命名

使用主动语态、动词优先：
- ✅ `creating-skills` 而非 `skill-creation`
- ✅ `condition-based-waiting` 而非 `async-test-helpers`

### 4. 跨引用其他技能

- ✅ 好：`必需子技能：使用 test-driven-development`
- ❌ 坏：`参见 skills/testing/test-driven-development`（不清晰是否必需）

### 5. 流程图使用

仅对以下使用流程图：
- 非显然的决策点
- 可能过早停止的过程循环
- "何时用A vs B"决策

绝不用于：
- 参考材料 → 表格、列表
- 代码示例 → 代码块
- 线性指令 → 编号列表

## 测试技能（RED-GREEN-REFACTOR）

### RED：编写失败的测试（基线）

在没有技能的情况下用子代理运行压力场景。记录确切行为：
- 他们做了什么选择？
- 他们使用了什么合理化借口（逐字记录）？
- 哪些压力触发了违规？

### GREEN：编写最小化技能

编写针对那些具体合理化借口的技能。不要为假设情况添加额外内容。

用相同场景运行现在有技能的测试。代理现在应该遵守。

### REFACTOR：堵住漏洞

代理发现了新的合理化借口？添加明确的反对。重新测试直到无懈可击。

## 技能抗合理化

### 明确堵住每个漏洞

不要只陈述规则——禁止具体的变通方法：

```markdown
在测试之前写代码？删除它。重新开始。

无例外：
- 不要保留它作为"参考"
- 不要在编写测试时"适配"它
- 删除意味着删除
```

### 应对"精神vs文字"争论

```markdown
违反规则的文字就是违反规则的精神。
```

### 构建合理化表格

```markdown
| 借口 | 现实 |
|--------|---------|
| "太简单不需要测试" | 简单代码也会出错。测试只需30秒。 |
```

### 创建红旗列表

```markdown
## 红旗 - 停止并重新开始
- 代码在测试之前
- "我已经手动测试过了"
所有这些意味着：删除代码。重新开始。
```

## 创建检查清单

- [ ] 名称只使用字母、数字、连字符
- [ ] YAML frontmatter 含必需的 `name` 和 `description`
- [ ] 描述以"Use when..."开头，包含具体触发条件
- [ ] 描述用第三人称
- [ ] 全文关键词
- [ ] 清晰的概述和核心原则
- [ ] 一个优秀示例
- [ ] 小型流程图（仅在决策非显然时）
- [ ] 快速参考表
- [ ] 常见错误章节
- [ ] 无叙述性故事
- [ ] 仅对工具或重型参考使用支持文件
