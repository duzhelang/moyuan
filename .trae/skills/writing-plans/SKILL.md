# 编写实现计划

## 复杂度门控（必须首先执行）

收到需求后，**在编写任何计划之前**，先按以下矩阵评估任务复杂度：

| 特征维度 | 直接执行（简单） | 调度执行（复杂） |
|----------|-----------------|-----------------|
| 涉及文件 | 仅单文件 | ≥2文件或跨模块 |
| 技术栈 | 单一栈且简单 | 跨栈或涉及数据层 |
| 风险等级 | 低风险（样式/文案/简单配置） | 中高风险（逻辑变更/数据/安全/需审查） |
| 审查需求 | 不需要 | 需要 |
| 文档影响 | 不涉及 | 需同步更新 |
| 复杂度 | 单步完成 | 多步骤/需规划 |

**判断规则：**
- 所有维度均落在「直接执行」列 → **简单任务**，不编写计划，直接告知：
  ```
  任务复杂度评估：简单任务（单文件/单栈/低风险/无需审查/无需文档同步）
  建议：直接执行，无需编写实现计划。
  ```
- 任意一个维度落在「调度执行」列 → **复杂任务**，继续执行下方完整计划编写流程。

## 概述

编写全面的实现计划，假设工程师对代码库一无所知且品味堪忧。记录他们需要知道的一切：每个任务要修改哪些文件、代码、测试、可能需要查阅的文档、如何测试。以小块任务的形式给出完整计划。DRY、YAGNI、TDD、频繁提交。

假设他们是熟练的开发者，但几乎不了解我们的工具集或问题领域。假设他们不太了解好的测试设计。

**开始时声明：** "我正在使用 writing-plans 技能来创建实现计划。"

**计划保存到：** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

## 范围检查

如果规范涵盖多个独立子系统，应该建议拆分为单独的计划——每个子系统一个。每个计划应该独立产生可工作、可测试的软件。

## 文件结构

在定义任务之前，规划哪些文件将被创建或修改以及每个文件的职责。

- 设计具有清晰边界和良好定义接口的单元。每个文件应该有一个清晰的职责
- 你最能把握能在上下文中容纳的代码，当文件更聚焦时你的编辑更可靠。优先选择更小、更专注的文件
- 一起变化的文件应该放在一起。按职责拆分，而非按技术层
- 在现有代码库中，遵循既有模式

## 小块任务粒度

**每个步骤是一个动作（2-5分钟）：**
- "编写失败的测试" - 一步
- "运行测试确保失败" - 一步
- "实现最小化代码使测试通过" - 一步
- "运行测试确保通过" - 一步
- "提交" - 一步

## 计划文档头部

**每个计划必须以此头部开始：**

```markdown
# [功能名称] 实现计划

> **给自动化工作者：** 使用 subagent-driven-development（推荐）或 executing-plans 技能来逐任务实现此计划。

**目标：** [一句话描述要构建什么]

**架构：** [2-3句话描述方案]

**技术栈：** [关键技术/库]

---
```

## 任务结构

```markdown
### 任务 N: [组件名称]

**文件：**
- 创建: `精确/路径/到/file.py`
- 修改: `精确/路径/到/existing.py:123-145`
- 测试: `tests/精确/路径/到/test.py`

- [ ] **步骤 1: 编写失败的测试**

```python
def test_具体行为():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2: 运行测试验证失败**

运行: `pytest tests/path/test.py::test_name -v`
预期: FAIL，提示 "function not defined"

- [ ] **步骤 3: 编写最小化实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4: 运行测试验证通过**

运行: `pytest tests/path/test.py::test_name -v`
预期: PASS

- [ ] **步骤 5: 提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: 添加具体功能"
```
```

## 禁止占位符

每个步骤必须包含工程师需要的实际内容。以下都是**计划失败**——永远不要写它们：
- "TBD"、"TODO"、"稍后实现"、"填写细节"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"
- "为上述编写测试"（没有实际测试代码）
- "类似任务N"（重复代码——工程师可能不按顺序阅读任务）
- 描述要做什么但不展示如何做的步骤（代码步骤需要代码块）
- 引用任何任务中未定义的类型、函数或方法

## 记住
- 始终使用精确的文件路径
- 每个步骤包含完整代码——如果步骤更改代码，就展示代码
- 精确的命令和预期输出
- DRY、YAGNI、TDD、频繁提交

## 自审

写完完整计划后，对照规范检查计划：

1. **规范覆盖：** 浏览规范中的每个章节/需求。能否指出一个实现它的任务？列出任何遗漏
2. **占位符扫描：** 在计划中搜索红旗——任何来自"禁止占位符"部分的模式。修复它们
3. **类型一致性：** 后续任务中使用的类型、方法签名和属性名是否与早期任务定义的一致？

## 执行交接

保存计划后，提供执行选择：

"计划已完成并保存到 `docs/superpowers/plans/<filename>.md`。两种执行选项：

1. **子代理驱动（推荐）** - 每个任务调度新子代理，任务间进行审查，快速迭代
2. **内联执行** - 在此会话中使用 executing-plans 执行任务，批量执行含检查点

哪种方式？"

如果选择子代理驱动：使用 subagent-driven-development
如果选择内联执行：使用 executing-plans
