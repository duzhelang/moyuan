# 系统化调试

## 概述

随机修复浪费时间并产生新bug。快速修补掩盖底层问题。

**核心原则：** 始终在尝试修复之前找到根因。症状修复是失败。

**违反此过程的文字就是违反调试的精神。**

## 铁律

```
没有先做根因调查就没有修复
```

如果还没有完成阶段1，就不能提出修复。

## 何时使用

对任何技术问题使用：
- 测试失败
- 生产环境bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用：**
- 时间紧迫（紧急情况让猜测变得诱人）
- "只需一个快速修复"看起来明显
- 你已经尝试了多次修复
- 之前的修复不奏效
- 你不完全理解问题

## 四个阶段

你必须完成每个阶段才能进入下一阶段。

### 阶段1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 注意行号、文件路径、错误码

2. **一致地重现**
   - 能否可靠触发它？
   - 确切的步骤是什么？
   - 每次都会发生吗？
   - 如果不能重现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 哪些改动可能导致此问题？
   - Git diff，最近的提交
   - 新依赖、配置更改
   - 环境差异

4. **在多组件系统中收集证据**
   - 在每个组件边界记录数据输入/输出
   - 验证环境/配置传播
   - 在每个层检查状态
   - 运行一次收集证据以显示在哪里出错

5. **追踪数据流**
   - 坏值从哪里来？
   - 什么用它调用了这个坏值？
   - 持续向上追踪直到找到源头
   - 在源头修复，而不是在症状处

### 阶段2：模式分析

**在修复之前找到模式：**

1. 找到相似的工代码
2. 与参考对比
3. 识别差异
4. 理解依赖

### 阶段3：假设与测试

1. 形成单一假设
2. 最小化测试
3. 在继续之前验证
4. 不理解时就说不理解

### 阶段4：实现

1. 创建失败的测试用例
2. 实现单一修复
3. 验证修复
4. 如果修复不奏效：如果 < 3次尝试，回到阶段1重新分析；如果 ≥ 3次失败，质疑架构

## 红旗 - 停止并遵循过程

如果发现自己在想：
- "快速修复一下，稍后再调查"
- "试试改X看看行不行"
- "加多个改动，跑测试"
- "跳过测试，手动验证"
- "很可能是X，让我修复它"
- "我不完全理解但这可能管用"
- "再试一次修复"（已经试了2+次）

**所有这些意味着：停止。回到阶段1。**

## 常见合理化借口

| 借口 | 现实 |
|--------|---------|
| "问题很简单，不需要过程" | 简单问题也有根因。过程对简单bug也很快。 |
| "紧急情况，没时间走过程" | 系统化调试比猜测-检查快得多。 |
| "先试这个，然后再调查" | 第一次修复设定了模式。一开始就做对。 |
| "一次性多个修复节省时间" | 无法隔离哪个起作用。产生新bug。 |
| "我已经看到问题了，让我修复它" | 看到症状 ≠ 理解根因。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 读错误、重现、检查更改、收集证据 | 理解是什么和为什么 |
| **2. 模式** | 找到工作的示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug解决、测试通过 |
