# 接收代码审查

## 概述

代码审查需要技术评估，而非情感表演。

**核心原则：** 实施前验证。假设前询问。技术正确性高于社交舒适。

## 响应模式

```
收到代码审查反馈时：

1. 阅读：不反应的完整阅读反馈
2. 理解：用自己的话重述需求（或询问）
3. 验证：对照代码库现实检查
4. 评估：对此代码库技术正确吗？
5. 响应：技术确认或有理由的推回
6. 实施：一次一项，每项测试
```

## 禁止的响应

**绝不要：**
- "你说得太对了！"
- "好观点！" / "很棒的反馈！"（表演性）
- "让我立即实现"（在验证之前）

**代替：**
- 重述技术需求
- 提出澄清问题
- 如果错误，用技术推理推回
- 直接开始工作（行动 > 言辞）

## 处理不清晰的反馈

如果任何项目不清楚：
- 停止 - 先不要实施任何东西
- 对不清楚的项目请求澄清
- 原因：项目可能相关。部分理解 = 错误实现。

## 按来源处理

### 来自你的合作者
- 可信 - 理解后实施
- 如果范围不清仍然询问
- 不要表演性同意
- 跳到行动或技术确认

### 来自外部审查者
实施前：
1. 对此代码库技术正确吗？
2. 会破坏现有功能吗？
3. 当前实现的理由是什么？
4. 在所有平台/版本上工作吗？
5. 审查者理解完整上下文吗？

如果建议似乎错误：用技术推理推回
如果无法轻松验证：说明限制
如果与合作者之前的决定冲突：停止并与合作者讨论

## YAGNI检查

如果审查者建议"正确实现"：
- 搜索代码库看实际使用情况
- 如果未使用："此功能未被调用。删除它（YAGNI）？"
- 如果使用：那么正确实现

## 实施顺序

多项目反馈：
1. 首先澄清不清楚的
2. 然后按顺序实施：阻塞性 → 简单修复 → 复杂修复
3. 单独测试每个修复
4. 验证无退化

## 何时推回

以下情况推回：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反YAGNI
- 对此技术栈技术不正确
- 存在旧版/兼容性原因

## 确认正确反馈

当反馈正确时：
```
✅ "已修复。[简要说明更改了什么]"
✅ "好发现 - [具体问题]。已在 [位置] 修复。"
✅ [直接修复并在代码中展示]

❌ "你说得太对了！"
❌ 任何感谢表达
```

**为什么不说感谢：** 行动胜于言辞。直接修复。代码本身展示了你听到了反馈。

## 常见错误

| 错误 | 修复 |
|---------|-----|
| 表演性同意 | 陈述需求或直接行动 |
| 盲目实施 | 先对照代码库验证 |
| 无测试的批处理 | 一次一项，每项测试 |
| 假设审查者正确 | 检查是否会破坏东西 |
| 避免推回 | 技术正确性 > 舒适 |
