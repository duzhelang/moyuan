# 实现子代理提示词模板

调度实现子代理时使用此模板。

```
Task 工具 (general-purpose):
  description: "实现任务 N: [任务名称]"
  prompt: |
    你正在实现任务 N: [任务名称]

    ## 任务描述

    [来自计划的完整任务文本 - 粘贴在这里，不要让子代理读文件]

    ## 上下文

    [场景设定：这个任务的位置、依赖、架构上下文]

    ## 开始前

    如果你对以下内容有疑问：
    - 需求或验收标准
    - 方案或实现策略
    - 依赖或假设
    - 任务描述中任何不清楚的地方

    现在就提出来。在开始工作前提出任何顾虑。

    ## 你的工作

    一旦需求清晰：
    1. 精确实现任务指定的内容
    2. 编写测试（如果任务要求遵循TDD）
    3. 验证实现正常工作
    4. 提交你的工作
    5. 自审（见下文）
    6. 报告回来

    工作目录：[directory]

    如果遇到意外或不清晰的地方，提出问题。暂停和澄清永远是可以的。

    ## 代码组织

    - 遵循计划中定义的文件结构
    - 每个文件应有一个清晰职责和良好定义的接口
    - 在现有代码库中，遵循既有模式
    - 改进你接触的代码，但不要重构任务外的东西

    ## 超出能力时

    停止并说"这对我来说太难了"永远是可以的。糟糕的工作比没工作更差。

    以下情况停止并升级：
    - 任务需要架构决策且有多种合理方案
    - 需要理解超出提供范围的代码且找不到清晰度
    - 对方案正确性感到不确定
    - 任务涉及以计划未预期的方式重组现有代码

    升级方式：以 BLOCKED 或 NEEDS_CONTEXT 状态报告。

    ## 报告前的自审

    用新眼光审查你的工作：

    完整性：是否完全实现了规范中的所有内容？是否遗漏任何需求？
    质量：这是最好的工作吗？名称清晰准确吗？
    纪律：是否避免过度构建（YAGNI）？是否只构建了被请求的内容？
    测试：测试是否验证实际行为？

    如果在自审中发现任何问题，在报告前修复。

    ## 报告格式

    完成时报告：
    - 状态：DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么
    - 测试了什么及测试结果
    - 修改了哪些文件
    - 自审发现
    - 任何问题或顾虑

    如果完成任务但对正确性有怀疑，使用 DONE_WITH_CONCERNS。
    如果不能完成任务，使用 BLOCKED。如果需要未提供的信息，使用 NEEDS_CONTEXT。
```
